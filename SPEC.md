## 2. Лендинг аренды `/arenda` — ГЕРОЙ (приоритет дизайна и разработки)

Задача экрана: за один-два экрана заставить массажиста/психолога сказать «да,
хочу вести здесь **регулярно**, на постоянной основе». Решение принимается по
фото, условиям и ощущению «это моё место надолго».

Блоки сверху вниз:

1. **Hero.** Заголовок-обещание в голосе бренда не про «аренду залов», а про
   «сделай это место своим регулярным». Крупное фото лучшего зала. Один CTA:
   «Закрепить за собой слот» / «Стать резидентом».
2. **Три зала — карточки.** Большой 70 м², Малый 30 м², Чайный 45 м². Для
   каждого: **фото** (по правилам бренд-бука), м², вместимость, оснащение,
   ставка с **акцентом на регулярную** (абонемент/месяц), почасовая — вторично;
   кнопка «Хочу вести здесь».
3. **Почему здесь** — аргументы атмосферы: дерево, свет, чай, тишина, забота. Для
   арендатора — как польза ЕГО клиенту каждую неделю + «пять опор» бренд-бука.
4. **Окно доступности** — какие **регулярные слоты** (дни/вечера) свободны под
   постоянную аренду. В V1 может быть упрощённо.
5. **Резиденты — кто уже постоянно ведёт у нас** — центральный блок доверия:
   постоянство продаётся через постоянство других. Из реальных данных Даниила.
6. **Форма заявки на аренду** (см. §4). Главная конверсия. Дублируется внизу.
7. **FAQ аренды** — как оформляется, что входит, депозит/правила.

## 3. Модель расписания «два CTA на слот»

Одна сетка расписания. Каждый слот — это запись с полем `cta_type`:

- `cta_type = "booking"` → кнопка «Записаться» → форма заявки на визит (наша).
- `cta_type = "external"` → кнопка «К специалисту» → внешняя ссылка
  (`external_url`, канал/сайт специалиста). Заявку ведёт сам специалист.

Это закрывает ответ «гибко: зависит от специалиста» без постройки двух систем.
**Никакой онлайн-оплаты в V1.**

**Открытый вопрос (см. PROJECT_GUIDE §10):** правило приоритета вечернего слота —
своё занятие vs аренда. До решения вечерние слоты помечаем флагом конфликта в
админке, чтобы не публиковать пересечения.

## 4. Формы

Общие правила для ВСЕХ форм:

- **Работают без JavaScript.** Нативный `<form method="POST">` на server-эндпоинт.
  JS — только прогрессивное улучшение (валидация на лету, без перезагрузки).
- **Идемпотентность через pre-generated ID.** Скрытое поле `idempotency_key`
  (UUID), сгенерированное при рендере формы. Повторная отправка того же ключа не
  создаёт второй лид. Ключ генерится ДО отправки, не из ответа.
- **Согласие на обработку данных** — чекбокс + ссылка на /privacy (как сейчас).
- **Антиспам:** honeypot-поле + время-на-форме, без CAPTCHA (CAPTCHA не решаем).
- **Без персональных данных в URL.** Только POST-тело.
- **Доступность — цель WCAG AA** (контраст из tokens уже под AA/AAA; семантика,
  alt, фокус, навигация с клавиатуры).

### 4.1 Заявка на постоянную аренду (главная)
Поля: имя, телефon/мессенджер, вид деятельности (массаж/психолог/коворкинг/…),
желаемый зал, **формат (регулярная / разовая)** — по умолчанию регулярная,
**регулярность** (какие дни недели и время постоянно), желаемый старт,
комментарий, согласие. После отправки: страница «спасибо» + понятный следующий
шаг. Цель Метрики: `arenda_lead`.

### 4.2 Заявка на визит (вторичная)
Поля: имя, телефон/мессенджер, интересующее направление, согласие. Цель:
`visit_lead`.

### 4.3 Регистрация на событие
Поля: имя, контакт, событие (скрытое поле = id события), кол-во мест, согласие.
Цель: `event_register`.

## 5. Модель данных (SQLite сейчас, Postgres-ready)

```
hall            id, name, area_m2, capacity, equipment, rate_hour, rate_day,
                rate_subscription, photos[], description
specialist      id, name, kind, bio, photo, external_url, is_resident
slot            id, hall_id, specialist_id?, weekday, time_start, time_end,
                title, cta_type ("booking"|"external"), external_url?,
                conflict_flag (bool), is_published
event           id, slug, title, description, datetime, hall_id, capacity,
                registered_count, is_published
lead            id (=idempotency_key UUID), type ("arenda"|"visit"|"event"),
                name, contact, payload(json), created_at, status, source_page
content_page    slug, title, body(md), hero_image, meta_description, updated_at
```

Сущности отражают границы: контент / расписание / лиды разделены.

## 6. Админка V1 — одна роль

Один администратор (Даниил/админ студии). Без мультиарендных ролей. Возможности:
- CRUD залов, специалистов, слотов расписания, событий, контент-страниц.
- Просмотр лидов (аренда/визит/события), смена статуса, экспорт.
- Флаг конфликта на пересекающихся вечерних слотах (до решения по приоритету).

Авторизация — простая (один аккаунт). **Создание аккаунтов и смену паролей
делает человек, не агент** (см. CLAUDE.md, запреты). Мультиролевая
самообслуживаемая админка — V2, в этой спеке её НЕТ.

## 7. Аналитика

- Яндекс.Метрика, счётчик **91052806** (перенос с текущего сайта).
- Цели: `arenda_lead` (главная — **заявка на постоянную аренду**), `visit_lead`,
  `event_register`.
- **Согласие на куки/обработку (152-ФЗ).** Метрика ставит куки → нужен баннер
  согласия + действующая /privacy. Самые приватные настройки по умолчанию; без
  персональных данных в URL.
- Schema.org **LocalBusiness** (адрес, телефон, часы 07:00–22:00) — базовая
  разметка; SEO-глубина отложена, но локальную разметку кладём сразу, она дешёвая.

## 8. Что НЕ входит в V1 (явно)

Онлайн-оплата · личные кабинеты · абонементы в кабинете · самообслуживаемая
админка специалистов · мультиролевой доступ · контент-маркетинг/блог · сложное
SEO. Всё это — V2 или параллельные проекты.
