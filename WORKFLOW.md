## 1. Принцип

Одна задача → одна ветка → один worktree → один пишущий агент → один PR.
Никаких двух пишущих агентов в одном файле одновременно (CLAUDE.md «правило
одного владельца шага»). Слияние в защищённые ветки делает **человек** (Даниил),
не агент (CLAUDE.md §2: деплой/релиз — человек).

## 2. Ветки

- `main` — защищённая. Только зелёный CI + ревью человека. Прямой push запрещён.
- Рабочая ветка задачи: **`<тип>/<ID>-<slug>`**, где
  - `<тип>` ∈ `ux | ui | brand | fe | be | copy | qa | infra | seo`
  - `<ID>` — из TASKS.md (напр. `T0-3`)
  - `<slug>` — короткий кебаб (напр. `arenda-hero`)
  - пример: `fe/T1-2-arenda-form`, `be/T2-4-leads-endpoint`
- Имя ветки = тип агента-владельца. Если задачу ведёт ux-architect — ветка `ux/...`.

## 3. Границы файлов (жёстко — это и есть защита от коллизий)

Каждый агент пишет ТОЛЬКО в свои каталоги. Пересечение = переразбить задачу.

| Агент | Пишет в | Не трогает |
|---|---|---|
| ux-architect | (артефакты IA — в PR-описание/`docs/`, кода нет) | `web/`, `api/` |
| ui-designer | `web/src/components/**`, `web/src/layouts/**`, макеты | `api/`, тексты |
| copywriter | `web/src/content/**`, микрокопи в компонентах по согласованию | код логики |
| frontend-engineer | `web/src/**`, `tests/web/**` | `api/src/**`, `design/tokens.json` |
| backend-engineer | `api/src/**`, `api/db/**`, `tests/api/**` | `web/src/**` |
| brand-guardian | НИЧЕГО (read-only, вето в ревью) | всё |
| qa-engineer | `tests/**` (планы/проверки), отчёты в PR | фичи |
| seo-specialist | ⛔V1: только по прямой просьбе — мета/Schema/редиректы | остальное |

`design/tokens.json` и `design/voice.md` — источники истины, правит человек.
`web/src/styles/**` генерится из tokens, вручную не редактируется.

## 4. Worktree на дорожку (bare-repo layout)

Параллельные дорожки — через `git worktree`, не через переключение веток в одном
рабочем каталоге (чтобы M1 не одна сессия на 8GB).

```
git worktree add ../mst-fe   fe/T1-2-arenda-form
git worktree add ../mst-be   be/T1-1-leads-endpoint
# по завершении:
git worktree remove ../mst-fe
```

Один активный worktree на агента. Закрывать по завершении задачи (привычка Даниила
«одна активная задача за раз»).

## 5. Цикл задачи (TDD)

1. Завести/взять `<ID>` из TASKS.md, создать ветку и worktree.
2. **RED:** написать падающий тест в `tests/web/` или `tests/api/`.
3. **GREEN:** минимальная реализация, тест зелёный.
4. Локально: `lint` + `test` + `build` зелёные.
5. Открыть PR по шаблону. Дождаться зелёного CI.
6. Если затронут визуал/текст — **brand-guardian ревью** (право вето на critical).
7. Слияние в `main` — **человек** после приёмки гейта стадии (PLAN.md).

## 6. Протокол передачи

Каждый агент завершает работу блоком **Передача** (Вход / Выход / Кому) — это
контракт пайплайна (дублируется в каждом `.claude/agents/*`). PR-описание ссылается
на этот блок: что пришло на вход, что отдаётся, какому агенту дальше.

## 7. Гейты человека (HITL)

Слияние gate-веток и релиз — только Даниил. Агенты НЕ выполняют (CLAUDE.md §2):
создание аккаунтов/смену паролей, права доступа, необратимое удаление, настройку
оплаты, деплой в прод без явного подтверждения. Эти шаги агент описывает в PR и
отдаёт человеку.
