# CLAUDE.md — Инварианты и оркестрация агентов

> Документ и инварианты, НЕ runtime-промпты (промпты живут в `prompts/`, в V1 пусто).
> Этот файл Claude Code читает в каждой сессии. Держать коротким и точным.

---

## 1. Инварианты проекта (нарушать нельзя)

1. **Определённость прежде магии.** V1 без runtime-LLM. Валидация/хэши/линт —
   детерминированный код.
2. **Идемпотентность через pre-generated ID.** `idempotency_key` (UUID) генерится
   при рендере формы, ДО отправки. Повтор ключа не создаёт второй лид. Не выводить
   ключ из ответа внешней системы.
3. **Формы работают без JavaScript.** Нативный POST. JS — только улучшение поверх.
4. **Двухфазность с человеком.** Перед записью во внешние системы — пауза и
   подтверждение Даниила. Не автопродолжать.
5. **Границы = каталоги.** `web/` ≠ `api/`; домены leads/schedule/events/content/
   admin не смешивать. Цвета/типографика — только из `design/tokens.json`.
6. **TDD RED→GREEN, приёмка Даниилом на каждом шаге.** Стадийная сдача (см. PLAN).
   Не бежать на следующий Gate без приёмки.
7. **Не расширять до валидации.** V2 (самообслуживаемая админка, оплата, кабинеты)
   не трогать, пока V1 не принял реальные заявки на аренду.
8. **Приоритет — дизайн + аренда.** Главная конверсия = заявка на аренду. SEO/
   маркетинг/ученическая воронка отложены.

## 2. Запреты по безопасности (делает человек, не агент)

Агенты НЕ выполняют: создание аккаунтов и смену паролей; ввод платёжных/
учётных данных; изменение прав доступа; необратимое удаление данных; настройку
оплаты/эквайринга; деплой в прод без явного подтверждения Даниила. Эти шаги агент
описывает и отдаёт Даниилу.

## 3. AI-«сотрудники» (субагенты) и ПРАВИЛА ВЫЗОВА

Авто-делегирование в Claude Code ненадёжно — **полагаемся на ЯВНЫЙ вызов**:
`Use the <agent> subagent to <task>`. Ниже — кто за что отвечает и когда звать.

### Карта ролей → когда вызывать

| Агент | Зовём когда | Не зовём для |
|---|---|---|
| `ux-architect` | нужны IA, карта экранов, пользовательский поток, вайрфрейм-логика | визуала, кода |
| `ui-designer` | визуальный дизайн экрана/компонента по токенам, макет лендинга | бэкенда, текстов |
| `brand-guardian` | ревью любого визуала/текста на соответствие бренд-буку (read-only) | написания кода |
| `frontend-engineer` | вёрстка/компоненты Astro, интеграция токенов, формы без JS | дизайн-решений, бэк-логики |
| `backend-engineer` | эндпоинты заявок, расписание (2 CTA), события, CRUD, схема БD | визуала, текстов |
| `copywriter` | тексты в голосе бренда (voice.md), микрокопи форм, контент страниц | дизайна, кода |
| `qa-engineer` | тест-планы, проверка форм/без-JS/мобайла/доступности перед Gate | написания фич |
| `seo-specialist` | ⛔ ОТЛОЖЕН (V2). В V1 — только Schema LocalBusiness, по прямой просьбе | в V1 не звать |

### Последовательность по умолчанию (для нового экрана/фичи)

```
ux-architect → ui-designer → brand-guardian (ревью) → frontend-engineer
                                  ↘ copywriter (тексты параллельно)
backend-engineer — параллельно, как только модель данных из SPEC ясна
qa-engineer — перед каждым Gate из PLAN
```

### Гейты человека (HITL)
- После `ui-designer` на лендинге аренды — приёмка визуала Даниилом (Gate M0).
- После `frontend+backend` на форме аренды — приёмка сквозного пути (Gate M1).
- `brand-guardian` — read-only, право вето на публикацию при нарушении токенов/голоса.

### Правило одного владельца шага
У каждого шага один ответственный агент, один вход, один выход, одно правило
передачи. Не запускать двух пишущих агентов на один и тот же файл одновременно.

### Параллельная работа через GitHub
Когда дорожки идут параллельно — следовать **WORKFLOW.md**: ветка на задачу
(`<тип>/<ID>-<slug>`), git worktree на дорожку, писать только в свои каталоги
(WORKFLOW §3), TDD RED→GREEN, PR по шаблону, CI зелёный, gate/release вливает
человек. ID задач — в TASKS.md.

### Протокол передачи (общий для всех агентов)
Каждый агент в своём файле заканчивает блоком **Передача** (Вход / Выход / Кому).
Это контракт пайплайна. Важно: субагент получает ТОЛЬКО свой системный промпт,
не этот CLAUDE.md — поэтому контракт продублирован в каждом `.claude/agents/*`.

### Runbook M0 (лендинг аренды — как собрать главной сессией, явные вызовы)
```
1. Use the ux-architect subagent to спроектировать структуру и поток /arenda.
2. Use the copywriter subagent to написать тексты лендинга (hero, залы, форма).
3. Use the ui-designer subagent to собрать макет /arenda на плейсхолдерах по tokens.
4. Use the brand-guardian subagent to проверить макет и тексты на бренд (вето при critical).
5. Use the frontend-engineer subagent to сверстать /arenda (TDD, формы без JS).
6. Use the qa-engineer subagent to проверить перед Gate M0.
7. → Приёмка визуала Даниилом (Gate M0). Дальше — только после приёмки.
```
backend-engineer на M0 не нужен (лендинг статичен); подключается на M1 (форма).

## 4. API-вариант (согласовать с R1)
По умолчанию (R1): бэк — **SSR-эндпоинты Astro внутри `web/src/pages/api/`**,
логические домены leads/schedule/events/content/admin сохраняются. Отдельный
каталог `api/` (FastAPI+Python) — запасной вариант, включаем только если упрёмся.
PROJECT_STRUCTURE показывает оба; при старте M0 зафиксировать один и убрать второй.
